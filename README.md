# UN-R13H ESS Alloy Study

自然言語で記述された自動車ブレーキ法規（UN-R13H 5.2.23 緊急制動表示灯）を題材に、
**LLM × Alloy（軽量形式手法）** によって自然言語要求仕様の **未定義領域（＝「行間」）** を形式的に検出する予備実験のリポジトリである。

本研究の核心は、**「要求仕様の全域化」をメタ要件として課し、その未充足領域を Alloy で反証列挙する** という新しい仕様検証アプローチの提案にある。

---

## 背景：仕様の「行間」という構造的問題

自動車OEMのECU向け自然言語仕様書には、明示されていない条件の組み合わせ — 「行間」 — が大量に存在する。これらは熟練エンジニアの暗黙知で補完されてきたが、SDV時代の仕様量は暗黙知による補完の許容範囲を超えつつある。見落としは実装・試験段階で初めて顕在化する。

従来の形式検証は **仕様内部の論理的整合性** は扱えても、**仕様から欠落している領域** は扱えない。欠落を検出するには、仕様そのものに対する **メタ要件（＝完全性の要請）** が必要である。

> **本リポジトリの問い：仕様の「行間」を、形式手法で自動的に発見できるか？**

---

## 研究の核心：要求は部分関数、コードは全域関数

| | 要求仕様（自然言語） | 実装（コード） |
|---|---|---|
| **数学的性質** | 部分関数 (Partial Function) | 全域関数 (Total Function) |
| **意味** | 定義域の一部にのみ値を返す | 全入力に値を返さねばならない |
| **帰結** | 入力の組み合わせに「行間」を許す | 未定義挙動は実装上許されない |

実装はあらゆる入力に対して物理的に何らかの状態に遷移するため、未定義入力を放置できない（cf. *input-enabledness*; Lynch & Tuttle 1989）。この **全域化ギャップ ＝ 暗黙知の宿る領域** を、形式手法で可視化することが本研究の目的である。

> **主張：** 要求仕様に「全域化」というメタ要件を課し、Alloy でその未充足領域を反証として自動発見する。暗黙知の属人化から、形式知の工程化へ。

---

## アプローチ：LLM × Alloy × Human の三者分業

LLM は「賢い検証者」ではなく、「自然言語 ⇄ 形式言語の高速翻訳器」として位置づける。**発見は Alloy、判断は人間、翻訳のみを LLM に委ねる** 三者分業が本研究の設計思想である。

| 役割 | 担当 | 得意 | 不得意 |
|---|---|---|---|
| **翻訳** | LLM | 言語間のパターン変換、意味次元の抽出 | 「何が欠けているか」の自律的問いかけ |
| **発見** | Alloy | 有界モデル検査による全域化ギャップの検出 | 埋めるべき値の妥当性判断 |
| **判断** | Human | ドメイン知識による全域化意図の決定 | 網羅的な組合せ探索、翻訳作業の量 |

### なぜ Alloy か

仕様の未定義領域を **網羅的に列挙** することが目的のため、命題を否定して反例モデルを列挙するモデルファインダ（Alloy / Kodkod）は、設計思想レベルで本研究の問題に直接対応する。SMT ソルバは個別反例の詳細検証フェーズで相補的に有用であるが、発見フェーズの第一選択としては、抽象度のミスマッチ（理論記号による余剰な推論）が課題となる。

## 実験対象

**UN-R13H 5.2.23 緊急制動表示灯（Emergency Stop Signal, ESS）**

急ブレーキ時にブレーキランプを高速点滅させ、後続車に追突の危険を警告する安全機能である。約200語の条文で構成され、OEMの自然言語仕様書に近い粒度を持つ公開文書であるため、予備実験の題材として選定した。題材自体は本研究の本質ではなく、自然言語仕様一般に適用可能な方法論を意図している。

---

## パイプライン

```
法規条文 ─→ LLMで意味次元抽出 ─→ LLMでAlloyコード生成 ─→ Alloy SATソルバ実行 ─→ 結果の解析・修正
(自然言語)                                                    ↑
                                                              └─ 反復改善ループ (Human-in-the-Loop)
```

直積空間の全パターンに対して、以下4つのステートを導出する：

| ステート | 意味 |
|---|---|
| **Permitted** | 法規条件に合致（点灯／消灯してよい） |
| **Forbidden** | 法規条件に違反（点灯／消灯してはならない） |
| **Conflict** | Permitted と Forbidden が同時に成立(矛盾) |
| **Unspecified** | いずれにも該当しない（未定義＝「行間」） |

**目標状態：Conflict = 0 かつ Unspecified = 0**（＝全域化の達成）

## ファイル構成

| ファイル | 状態数 | 概要 |
|---|---|---|
| `01_UNR13H_ESS_Initial.als` | 576 | 初期モデル。LLMが生成したコードをベースに、意味次元の直積空間を全列挙して4分類 |
| `02_UNR13H_ESS_Hysteresis.als` | 1,152 | 単一の `prev`（前回信号状態）次元を追加。中間帯の挙動を明示化 |
| `03_UNR13H_ESS_Prioritize_by_or.als` | 1,152 | 点灯トリガーの OR 回路化。個別条文の Forbidden が出ても、いずれかのトリガーが有効なら上書き（Conflict = 0 達成） |
| `04_UNR13H_ESS_Predicate_Based_Fixed.als` | **4,608** | 5.2.23.1 / 5.2.23.2(a) / 5.2.23.2(b) の3条件それぞれに独立の `prev` 次元を導入。**述語ベース**（全称量化）で再実装し、真の完全閉包（Conflict = 0 ∧ Unspecified = 0）に到達 |

## 意味次元

約200語の条文から LLM（Gemini Pro 3.0）で抽出した意味次元：

| 意味次元 | 値域 | 根拠条文 |
|---|---|---|
| 主制動装置 | 作動 / 非作動 | 5.2.23 |
| 減速度域 | < 2.5 / 2.5–6.0 / ≥ 6.0 [m/s²] | 5.2.23.1 |
| 制動要求予測 | < 2.5 / 2.5–6.0 / ≥ 6.0 [m/s²] | 5.2.23.2(a) |
| 車速域 | ≤ 50 km/h / > 50 km/h | 5.2.23.2(b) |
| ABS状態 | ABSフルサイクリング / ABS非制御 | 5.2.23.2(b) |
| 設計パターン | 減速度条件のみ / +減速予測 / +ABS / +両方 | 5.2.23.2 |
| ESS信号 | 点灯 / 消灯 | — |

2 × 3 × 3 × 2 × 2 × 4 × 2 = **576 状態**

その後の拡張：
- `02`：単一の `prev` 次元追加で **1,152 状態**
- `03`：OR 結合導入（同 1,152 状態、Conflict = 0 達成）
- `04`：3 条件独立の `prev` 次元 + 述語ベース実装で 576 × 2³ = **4,608 状態**（真の完全閉包達成）

## 発見された3つの「行間」

### 課題 a：禁止の NOT の行間

条文「発動及び停止は…主制動装置の作動によってのみ行う」に対し、LLM は「非作動時は点灯禁止」のみをコード化し、その裏にある暗黙の許可「非作動時は消灯してよい」を自律的に導出できなかった。非作動 ＋ 消灯が `Unspecified` に分類され、**Alloy の Gap 検出 → LLM 解析 → 人間の判断**によって「消灯許可」条件を追加し解消。

> **行間パターン：** 「禁止のNOT＝許可」 という自然言語特有の暗黙構造。

### 課題 b：OR 条件の行間

5.2.23.1（実測減速度）、5.2.23.2(a)（減速度予測）、5.2.23.2(b)（ABS条件）は **独立したトリガーとして OR 結合** される構造を条文が暗示するが、明示はされていない。LLM はこれらを独立条件（事実上のAND）として実装した結果、「減速度 < 2.5（禁止） vs ABS作動（許可）」のような Conflict が **77件** 発生。**Alloy の Conflict 検出 → LLM 解析 → 人間の判断**によって OR 回路として合成し、Conflict = 0 に到達。

> **行間パターン：** 複数条件の論理結合（OR/AND）の暗黙的指定。

> 本ケースは重要な示唆を含む：LLM が安易に OR 結合とせず AND 結合とした結果、**Alloy の Conflict 検出が正しい要求構造への修正を促した**。LLM が論理結合を勝手に倒すと設計リスクの見過ごしが発生し得る。Conflict 検出 → 人間の判断 → 修正、という Human-in-the-Loop の歯止めをパイプラインのどこに置くかが研究課題となる。

### 課題 c：ヒステリシスの独立性

LLM は条文の中間帯（2.5–6.0 m/s² など）に気づいてヒステリシス（前回状態 `prev` 次元）を付加し、**要件充足と誤判断して 1,152 状態で閉じた**。しかし本来は 5.2.23.1（実測減速度）、5.2.23.2(a)（減速度予測）、5.2.23.2(b)（ABS条件）の **3 条件それぞれが独立にヒステリシスを持つべき** であり、LLM の Alloy 結果解析ではこの状態分離の必要性に気づけなかった。人間が発見して 3 条件独立の `prev` 次元を導入し、述語ベース実装（`04_UNR13H_ESS_Predicate_Based_Fixed.als`）で 576 × 2³ = **4,608 状態** に拡張して真の完全閉包に到達。

> **行間パターン：** 一見もっともらしい解（単一ヒステリシス）が、実は不完全であるというメタな見落とし。**LLM 単体では「自分が見落としたこと」に気づけない**ことを示す典型例であり、本研究が三者分業（特に Human による独立性判断）を必要とする最も強い根拠となるケース。

## 先行研究との位置づけ

本研究は **「LLM × 仕様の不完全性発見」** という空白象限を切り拓くことを目指す。

|  | 仕様⇔コードの整合性検証 | 仕様の不完全性検証 |
|---|---|---|
| **人間による形式手法展開** | CBMC, ACSL/Frama-C, Dafny | TLA+, Goal Trees and Patterns (Aoki et al. 2016) |
| **LLM による形式手法展開** | VeCoGen, Clover, spec2code | **本研究** |

先行研究の到達点は、仕様検証は人手で、LLM活用はコード検証で進展してきた。本研究の貢献は、LLM の力を **仕様側の不完全性発見** に向ける点にある。

## リサーチクエスチョン

予備実験から導出された3つのリサーチクエスチョン：

### RQ1：プロンプト設計 — LLM に暗黙を問わせる
LLMが見落とす行間パターン（粒度誤認、OR/AND誤り、暗黙許可など）の体系化と、行間への気づきを促すメタ認知プロンプトの設計。

### RQ2：パイプライン設計 — Alloy 出力を LLM へ還流
Alloy の `Unspecified` / `Conflict` 結果を LLM が自己修正プロンプトに取り込む機構の設計と、反復ループの収束条件・終了条件の理論化。

### RQ3：Human-in-the-Loop（HITL）配置 — 人間の判断点をどこに置くか
パイプライン各ステップに対する「人間が必要 / 自動化可能」の判定基準、および熟練エンジニアの暗黙知との照合による HITL 有効性評価。

> **位置づけ：** 本リポジトリは予備実験段階のドラフトであり、各 RQ の方法論は今後の研究で精密化していく。
